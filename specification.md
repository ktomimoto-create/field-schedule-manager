# フィールド行動予定管理システム 仕様書 (specification.md)

本書は、フィールド行動予定管理システムにおけるスタッフ管理、同期処理、および各画面での表示に関する仕様を定義します。

---

## 1. スタッフの分類とマスタ仕様

システムに登録されるスタッフは、大きく**「FTS社員」**と**「外注メンバー（協力会社社員）」**に分類されます。

### 1.1 FTS社員
* **特徴**: 社員番号（`employee_code`）および本物のメールアドレス（`@fts.co.jp` 等）を持ちます。
* **アバター解決**: Microsoft Entra ID (Azure AD) のプロファイルと連携し、社員番号またはメールアドレスをキーとして自動的にプロフィール画像（アバター）を取得します。
* **自動同期**: 「Microsoftアカウント同期」機能の対象となり、未設定のメールアドレスや社員番号が自動で補完されます。

### 1.2 外注メンバー（マスタ外対応者）
* **特徴**: 協力会社のメンバーであり、FTSの社員とは異なります。
* **判定基準**: 
  - 氏名が `FE`, `SF`, `FR` プレフィックスから始まる（例: `SF濱田`、`FR岡崎`）
  - または、デフォルトコース番号が `90` 以上
* **メールアドレス・社員番号**: 
  - メールアドレスはダミー（例: `hamada@example.com`）を原則保持します。
  - 社員番号 (`employee_code`) は `null` (未設定) とします。
* **アバター解決**: Microsoft アカウントとの紐付けを行わないため、画面上は名前の頭文字を用いた**「イニシャルバッジ」**（例: `S` や `F`）で表現されます。
* **自動同期**: **同期処理から完全に除外（自動スキップ）**されます。名前の部分一致などによって、同姓のFTS社員（例: 濱田正貴、岡﨑勝二）と誤って紐付けられることを防ぎます。

## 2. 自動同期仕様 (Microsoftアカウント同期)

マスタ管理画面の「Microsoftアカウント同期」を実行した際の同期仕様は以下の通りです。

1. **除外ガード**:
   - 対象スタッフが外注メンバー（`FE`, `SF`, `FR` で始まる、またはコース番号90以上）である場合、同期処理を自動的にスキップします。
2. **既存アドレスの保護**:
   - すでにダミーアドレス（`@example.com`）以外の本物のメールアドレスが登録されているスタッフは、上書きを防ぐため同期処理をスキップします。
3. **名寄せキー優先順位**:
   - **社員番号による完全一致**: スタッフに社員番号が登録されている場合、外部プロファイルの `employee_id` との完全一致で紐付けます（最優先）。
   - **名前による一致**: 社員番号が未登録、または一致しない場合、スタッフ名からプレフィックスを除去したクリーンな苗字と、外部プロファイルの `display_name`（スペース除去・表記揺れ正規化後）を突合します。
     - ただし、同姓同名の被りを防ぐため、完全一致するプロフィールが「唯一（1件のみ）」存在する場合のみマッチと判定します。複数マッチした場合は警告を出力してスキップします。
     - ※ 「フーギー（ナルマンダフ・フスレンバヤル）」などの特殊名については個別の例外マッチングルールを適用します。
4. **同期内容**:
   - マッチした外部プロファイルのメールアドレス（小文字化）および社員番号（未設定の場合のみ）を、Supabaseの `staff` テーブルに書き込みます。

## 3. UI表示およびソート仕様

### 3.1 マスタ管理画面 (MasterManagementView)
* **スタッフ一覧**:
  - 氏名列には、**アバター画像（またはイニシャルバッジ） ＋ 登録氏名（プレフィックスを含むフルテキスト、例: `SF濱田`）**を表示します。
  - 外注メンバーなどアバター画像がないスタッフは、登録氏名の頭文字を表示する丸型バッジとなります。
  - FTS社員などを表示する際、画面上で名前を省略せず、マスタに登録されている氏名でフル表示します。

### 3.2 当日行動予定表 (TimelineView)
* **スタッフボードのソート順**:
  - 画面に表示されるスタッフボード（各スタッフごとの列）は、デフォルトコース（`default_course`）の**数値昇順**（例: 1, 2, ..., 26, 27, 90, 91...）で左から右へ並び替えて表示します。
  - コース番号が数値に変換できない、または未設定のスタッフは、リストの末尾に配置されます。

### 3.3 その他の画面での表示形式（既定）
* **当日・月間行動予定表**: アイコン ＋ 苗字（getShortName による変換後）
* **行動予定**: 苗字のみ

## 4. 月間予定表（CalendarView）における予定ステータス仕様

### 4.1 予定のステータスとデザイン表現
すべての予定セルは視認性を考慮して**白背景（通常背景）**を基本とし、手動で割り振られたステータスに応じて以下のアクセントを適用します。

* **通常（フリー）予定 (free)**:
  - 新規作成時や、ステータスが手動割り当てされていない状態の標準予定です。
  - 背景色：白（無色のセル）
  - 左端のアクセントバー：なし
* **確定予定 (confirmed)**:
  - 背景色：白
  - 行の先頭セル（`type` 列のセル）の左端に、太さ 4px の **グリーンのアクセントバー (#10b981)** を表示して確定状態を示します。
* **仮予定 (draft)**:
  - 背景色：白
  - 行の先頭セル（`type` 列のセル）の左端に、太さ 4px の **オレンジのアクセントバー (#f59e0b)** を表示して仮の状態を示します。
* **キャンセル (cancelled)**:
  - 背景色：灰色 / 取消線
  - 仕様：予定がキャンセルされた際、対応予定者（`staff_id`, `staff_name`）およびコース番号（`course`）を自動的に空白（クリア）にし、区分（`division`）を `未定` に書き換えます。これにより二重手配を防ぎます。
  - 仮想フリー行の復活：キャンセルされた予定は「予定なし」とみなされるため、元の対応者スタッフは本来のコース順（上部）に仮想フリー行（空き枠）として再び復活して表示されます。
  - ソート配置：カレンダーおよび予定表（グリッド）画面において、キャンセルされた予定（空白化された実データ）は常にリストの最下部に移動して表示されます。

### 4.2 ステータスのワンクリック切り替え
* 予定セルを右クリックした際に表示されるカスタムコンテキストメニューから、現在のステータス以外のトグル（例: **「予定を【フリー】に変更」**、**「予定を【確定】に変更」**、**「予定を【仮】に変更」**）をワンクリックで実行できます。
* 選択したステータスは即座にデータベース（Supabase）に反映され、再読み込みなしで画面上が更新されます。

### 4.3 予定追加・編集UI（サイドバー方式）
* **スライドイン表示**:
  - 予定のセルをダブルクリックするか、右クリックから「詳細を編集」を選択、または「新規予定の追加」を実行した際、画面の右端から幅 480px のサイドバーが滑らかにスライドインして表示されます。
* **高級ガラスモルフィズムデザイン**:
  - サイドバーは半透明のガラスぼかし背景（`backdrop-filter: blur(20px)`）を採用し、背後のカレンダーを美しくぼかして透過させつつ、情報の視認性を確保します。ライトテーマ・ダークテーマの双方に完全に調和する透過デザインです。
  - 最上部にグラデーション（プライマリからアクセント）の細いアクセントラインを配置し、高級感のあるUIにします。
* **操作性の向上**:
  - 背景の暗幕（モーダル遮蔽）を排除し、サイドバー表示中も背後のカレンダーのスクロールや閲覧、他日付セルの確認ができるように操作を素通し（クリック素通し）します。
  - フォーム内のヘッダーおよび最下部の操作ボタン（保存、キャンセル、削除）は常に固定され、入力項目部分だけが独立してスクロールするため、操作がしやすくなっています。

### 4.4 予定の作成・インポート操作と適用範囲
予定の新規追加およびデータインポート用のボタンは、システム上の適用範囲（`is_transferred`：予定表グリッドへの移行状態）に応じて、それぞれの画面内（各ビューのヘッダー領域）に分離して配置されています。

* **月間予定表（カレンダー）内のボタン**:
  - 配置箇所：日付並列カレンダーグリッドのヘッダー（`matrix-header`）内の日付操作コントロールの右隣。
  - **「スプレッドシートから貼り付け」**: コピペされたTSVデータをインポートします。インポートされたデータは、デフォルトで **`is_transferred: 0` (月間予定表にのみ適用)** として保存されます。
  - **「予定を追加」**: 新規作成モーダルを開きます。ここから追加されたデータは、デフォルトで **`is_transferred: 0` (月間予定表にのみ適用)** として保存されます。
* **予定表グリッド内のボタン**:
  - 配置箇所：予定表グリッドのヘッダー（`grid-view-header`）内の日付操作コントロールの右隣（管理者権限限定）。
  - **「予定を追加」**: 新規作成モーダルを開きます。ここから追加されたデータは、デフォルトで **`is_transferred: 1` (予定表グリッドにのみ追加/即時反映)** として保存されます。

## 5. 画面レイアウトとスクロール仕様 (Flexbox 100vh固定)

システム全体は、ブラウザウィンドウ自体の縦スクロールが発生しない**ダッシュボードレイアウト**として設計されています。

### 5.1 画面高さの完全フィットとコンパクト設計 (Flexbox)
* **アプリコンテナ**: `height: 100vh; overflow: hidden;` により、ブラウザ全体の縦スクロールを完全に禁止します。
* **ヘッダーのコンパクト設計**:
  - `App.css` を正しくインポートし、以前巨大化（100px）していたアバター画像を直径 24px の円形に縮小表示。
  - ヘッダーのパディングを `0.4rem 1.5rem`、要素間 `gap` を `0.75rem` に削減。
  - ロゴのフォントサイズを `1.15rem`、タブボタンのパディングを `0.35rem 0.75rem` (フォントサイズ `0.82rem`)、アクションボタンのパディングを `0.35rem 0.75rem` (高さ `32px` 固定) にそれぞれ縮小。
  - アイコンのみのボタン（ダーク切替、更新など）の幅を 32px に固定。
  - これにより、デスクトップ解像度ではすべてのヘッダー要素が1段（1行）にすっきりと収まるように最適化されます。
* **ヘッダーの折り返し保護**: 画面幅が極端に狭まった場合でも、各要素が潰れないように `flex-shrink: 0` を維持しつつ、自動折り返しが発生します。
* **メインエリア (`main`)**: ヘッダーの高さが変動しても、残りの高さを自動的に100%占有します（`flex: 1; min-height: 0`）。
* **各ビューコンテナの自立スクロール化**: 画面全体のスクロールが発生しないため、メインエリア内に表示される各ビューのルートコンテナ（マスタ管理 `.master-mgmt-container`、集計分析 `.analytics-container`、変更履歴 `.audit-log-container` 等）に `flex: 1; min-height: 0; overflow-y: auto;` を設定し、コンテンツが画面高さを超えた場合にそれぞれのビュー内で縦スクロールが可能となるように制御します。

### 5.2 各予定表ビューの内部スクロールと操作コントロール
* **日付操作・予定アクションの常時表示とすっきり化**:
  - カレンダーおよびグリッドビューのヘッダー領域（`matrix-header`、`grid-view-header`）は、画面全体のスクロールが無いため常にグローバルヘッダーの真下に固定して表示されます。
  - カレンダーの見出しの下にあった操作ガイドおよびステータス凡例文言（「仮予定 確定予定 ...」）は非表示（削除）にし、ヘッダー領域を極限まですっきりさせました。
  - カレンダーの月移動ボタンは、固定の「前の月」「次の月」という表示から、現在表示中の月を基準に前後の月名（例: **「5月」**、**「7月」**）を動的に算出して表示する仕様に変更されました。
  - 各ボタンの並び順は、左から「**[前の月名] [次の月名] [日付指定] [本日] [貼り付け] [予定を追加]**」の順に最適化され、各テキストもそれぞれ「日付指定ジャンプ」➔「日付指定」、「今日」➔「本日」に簡素化されました。
* **テーブルの自立スクロール**:
  - カレンダーのテーブルラッパー（`.matrix-table-wrapper`）およびグリッドのテーブルラッパー（`.grid-table-wrapper`）のみが、画面領域に収まった状態で自立してスクロール（`overflow: auto`）します。
  - これにより、縦スクロールしても日付操作・予定アクションボタン群が隠れてしまうことがなく、かつ曜日別の「横スクロールバー」も常に画面最下部に表示され、月曜〜日曜日のカレンダー表示を快適に横スクロールできます。
